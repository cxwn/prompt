# Step 8｜解析模板并编写交底书（核心）

## 8.1 模板解析与写作约束（不可突破）

- 必须完整解析并严格遵循用户提供的交底书模板；
- 不得：
  - 增删章节；
  - 调整章节顺序；
  - 重命名标题。
- 所有技术内容必须：
  - 具备明确工程实现方式；
  - 可被本领域技术人员直接复现；
  - 可拆解为权利要求中的技术特征。

---

## 8.2 技术写作总原则（CN 审查导向）

交底书中涉及的**技术定义、规则描述、数学表达、控制逻辑**，统一遵循以下原则：

> **能不用形式化就不用；必须用时，做到最简单、最确定、最可复核。**

---

## 8.3 技术概念定义要求

- 关键技术对象首次出现时必须定义；
- 定义内容仅限：
  - 技术对象是什么；
  - 属于哪个系统或模块；
  - 在方案中的具体作用。
- 禁止：
  - 仅描述技术效果；
  - 使用抽象、泛化或类比性表述；
  - 使用“智能、动态、自适应”等未限定词汇。

---

## 8.4 数学与规则表达使用边界（强约束）

### （1）允许使用的场景（必须满足其一）

仅当以下内容构成**核心技术手段**时，才允许使用数学或形式化表达：

- 明确的计算或判定公式；
- 明确的控制、调度或决策规则；
- 输入与输出之间的确定性关系。

### （2）禁止使用的场景（直接判负面）

- 用数学表达替代工程说明；
- 引入概率模型、统计推断或学习过程；
- 为体现复杂性而引入高阶函数或多层推导；
- 无法给出明确工程含义的抽象建模。

---

## 8.5 LaTeX 表达最低合规要求

凡使用公式，必须：

1. 使用 LaTeX 语法，并**直接嵌入 Markdown 正文**；
2. 给出所有符号的明确含义；
3. 说明符号的取值范围或工程来源；
4. 明确公式适用的系统条件或边界。

公式的唯一目的应当是：  
**限定技术手段的实施方式，而非解释技术效果。**

---

## 8.6 数学表达示例（粒度上限）

```markdown

系统根据采集的运行参数计算判定值 $S$，其计算方式如下：
$$
S = \sum_{i=1}^{n} w_i \cdot x_i
$$
其中，$x_i$ 为第 $i$ 项运行参数，$w_i$ 为对应权重，$n$ 为参数数量。当 $S$ 超过预设阈值 $T$ 时，执行对应的控制操作。

```

该示例代表**CN 审查允许的最大复杂度**，不得在非必要场景超出。

---

## 8.7 工程等价映射要求（强制）

交底书中必须形成如下可复核对应关系：

- 系统行为 → 工程处理步骤；
- 参数 → 可获取、可配置的系统变量；
- 公式 / 规则 → 核心技术手段；
- 条件 / 阈值 → 运行约束；
- 技术效果 → 可验证的工程结果。

任一数学或规则表达，均必须能映射至具体工程实现。

---

## 8.8 流程图与架构图约束（Mermaid）

- 涉及系统结构或方法步骤时，必须提供图示；
- 图示必须使用 Mermaid 语法，并**直接嵌入 Markdown 正文**；
- 图示仅用于：
  - 限定模块边界；
  - 明确流程顺序；
  - 说明数据或控制关系。
- 禁止通过图示引入正文中未定义的技术内容。

---

## 8.9 一致性与止损校验（隐式执行）

在输出交底书前，必须确认：

- 数学表达未引入新的技术问题；
- 数学表达未扩大保护范围；
- 去除公式后，技术方案仍可完整表述。

不满足上述条件的公式，应删除。

---

## 8.10 输出与流程推进

- 技术交底书统一以 Markdown 输出；
- 若支持文件导出，可同步提供：
  - `.docx / .doc` 文件；
  - 或可下载文件链接。

用户确认或直接输入下一步提示词后，进入 Step 9。
